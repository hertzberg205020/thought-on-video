# 影片心得 (統一介面架構: 一個 API 統一前後端開發)

影片中雖未明言，但根據其描述的「統一介面架構」特性推測，其底層實作應是基於 GraphQL。簡言之，GraphQL 賦予前端類似執行 SQL 查詢的能力，讓開發者能精準地從後端獲取所需資料。透過這種方式，後端被抽象化為一個靈活的資料服務層，有效實現前後端開發的解耦。下文我將採用 Pattern 的結構，對此解決方案進行深入分析。

## Pattern 分析

### Context

前端與後端的關注點存在本質差異 —— 後端按照領域邏輯組織資料，關注業務規則的正確性與資料一致性；前端按照畫面需求組織資料，關注使用者體驗與視覺呈現。兩者的演進節奏不同：後端受業務規則驅動，變更頻率較低且穩定；前端則隨 UI 設計調整，變更極為頻繁。

### Problem

如何讓前後端團隊在「資料如何提供給消費者」這件事上解耦，使雙方能夠依照各自的節奏獨立演進，同時維持資料傳輸的效率？

### Forces

#### Force 1：前後端的變更節奏不對稱

後端受業務規則驅動，變更頻率低且可預期。前端受 UI 設計、市場策略驅動，變更頻率高且難以預期。在 RESTful API 架構下，由後端團隊定義每個 endpoint 的回傳結構。每當前端需要新的資料組合時，就必須與後端協作，請求調整或新增 Endpoint。這種緊密的耦合，使得整體開發效率受限於變更節奏較慢的一方（通常是後端）。這並非開發能力的優劣問題，而是兩者服務的核心目的不同，導致開發步調存在本質上的差異。

#### Force 2：資料的組織原則不同

後端按**領域實體**切割資料（User, Order, Product），追求正規化與一致性。前端按**畫面元件**切割資料（這張卡片要顯示什麼、這個列表要顯示什麼），追求符合使用者認知的資訊組合。這兩種切割方式幾乎不可能對齊，因此「重新組裝」是必然的。問題在於：由誰決定怎麼組裝，以及組裝邏輯在哪一層執行。

#### Force 3：資訊不對稱造成組織瓶頸

前端雖然最清楚畫面需求，卻對 API 回傳內容缺乏控制權；後端雖然主導了資料結構，卻難以預知各個前端頁面的具體呈現需求。在 RESTful API 架構下，每次需求變更都意味著一次跨團隊溝通。**溝通成本將隨著專案規模、團隊人數及複雜度的提升而呈線性增長。**

#### Force 4：資料的圖狀本質 vs. 端點的樹狀結構

現實世界的資料關聯是圖狀的，但 RESTful API 把它切成一個個獨立的端點，每個端點是一棵淺樹。當前端需要橫跨多個實體的資料時，往往被迫發出多次請求，再由前端自行組裝資料。這是架構的結構性限制，不是流程改善可以解決的。

#### Force 5：網路來回的物理代價

每次 HTTP 請求都存在固有的延遲成本。請求多個 Endpoint 意味著多次往返 (Round-trip)，這在行動網路或高延遲環境下會嚴重影響使用者體驗，屬於網路通訊層面的硬性限制。

### Solution

引入一個**自助式資料存取層（Self-Service Data Access Layer）**，具體做法：

1. **由後端定義 Schema（型別系統）**：使用強型別描述所有資料類型與關聯關係。這份 Schema 作為雙方共享的契約，不僅反映了後端的領域模型，也將「定義資料邊界」的權利保留在後端。

2. **由前端撰寫 Query（查詢）**：前端透過宣告式語言定義所需的資料結構，發送至單一端點後，系統將自動組裝並回傳格式完全符合需求的結果。「要什麼資料」的決定權交給前端。

3. **組裝邏輯自動化**：resolver 層自動處理參照解析、關聯走訪、欄位篩選，不需要人工為每種需求撰寫專屬的 endpoint。

**這個解決方案的本質是：用一個技術契約（schema）取代反覆的人際溝通，把原本需要跨團隊協調才能完成的事情，轉化為前端在契約範圍內的自助操作。**

換句話說，GraphQL 做的事情是：

- **識別出「哪些協調是重複的、可規格化的」**——每次前端需要一組新的資料組合，溝通的模式其實都一樣：「我要 A 實體的 x 欄位、B 實體的 y 欄位、它們之間的關聯是 z」。這種溝通具有固定的結構。
- **把這個結構化的溝通轉化成一套協定（protocol）**——schema 定義了可以存取哪些資料，query language 定義了怎麼查詢。
- **使用技術代替溝通來執行這個協定**——resolver 自動完成原本需要後端工程師手動撰寫 endpoint 的工作。

團隊中許多反覆發生的溝通協調，可透過技術手段落實結構化與自動化，從而將人力從重複性的瑣事中釋放出來。

### Resulting Context

#### 正面效果

- **前後端解耦的程度從「流程層面」躍升至「架構層面」。** 團隊不再依賴頻繁溝通來同步需求，只要 Schema 涵蓋所需，前端即可自行調整 Query。前端的變更不再觸發後端的變更流程。
- **跨團隊溝通的頻率大幅降低。** 溝通集中在「schema 的演進」上——什麼時候新增類型、什麼時候廢棄欄位。這類變更的頻率遠低於個別畫面的需求變更。
- **多樣化的 API 消費端（如不同平台的 App 或 Web）不再成為開發瓶頸。** 不需要後端為每個團隊客製化 endpoint。
- **資料傳輸效率提升。** 單一請求、精準取得所需欄位，under-fetching 和 over-fetching 同時被解決。
- **Schema 成為活的文件。** 前後端對「系統有哪些資料、如何關聯」有了精確且始終同步的共同理解，減少了認知差異。
- **用技術手段解決組織內「重複性的、可結構化的跨團隊協調」的問題，將其轉成一套技術協定，進而打破團隊間的結構性依賴。**

#### 需要進一步處理的 Forces（可再套用其他 Pattern）

- **後端複雜度增加。** 前端的自由度意味著後端要能處理任意結構的查詢。
- **安全性與效能控制。** 查詢的決定權在前端，後端需要防範惡意或過度複雜的查詢。
- **快取策略需重新設計。** REST 的 URL-based 快取不再適用。
- **Schema 演進需要治理。** Schema 成了全系統的共享契約，它的變更影響所有消費者。需要 **Schema Governance** 的流程——版本管理、廢棄策略、向後相容性規範。這是一個新的組織問題，但它的規模和頻率比原本的「每個畫面都要協調」小得多。
- **學習成本。** 團隊需要理解新的概念、工具鏈、除錯方式。短期內生產力會下降。

## 個人的見解

我認為 GraphQL 的核心價值，在於將複雜的組織溝通問題轉化為可規範的技術框架，透過「統一介面資料結構模型」解決前端介面需求多變時的協作問題。

然而，對於前端 UI 變動頻率不高的專案，或是以 Feature Team 為核心組織架構的團隊，導入 GraphQL 的邊際效益未必顯著。
以我個人日常開發的製造業內部系統為例，多數專案由我一人同時負責網頁前端頁面與後端 Web API，系統的核心在於穩定地執行生產流程，並確保嚴謹的資料一致性。這類系統的 UI 呈現通常高度遵循標準作業流程 (SOP)，需求相對穩定且明確。

因此，在領域模型如此清晰的情境下，我傾向維持 RESTful API 架構，並採用 Contract-First（契約優先）的開發模式：初步定義 OpenAPI Specification 與資料庫 Schema 後，直接從規格生成帶有強型別的 TypeScript 介面 / JSDoc 與 Mock Service，供前端先行建構 UI。若開發過程中發現資料缺漏，我也能即時回頭修正 API 規格或調整資料表結構，迭代過程更為高效。

當專案規模擴大至多人協作時，若專案性質屬於 UI 需求相對固定、變動性低的類型，API 規格的異動頻率自然不高。在溝通成本尚能受控的前提下，維持現有的 RESTful 架構會更為務實，未必需要引入 GraphQL 來增加技術架構的複雜度。
