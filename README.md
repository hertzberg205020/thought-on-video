# 影片心得 (統一介面架構: 一個 API 統一前後端開發)

影片中雖未明言，但根據其描述的「統一介面架構」特性推測，其底層實作應是基於 GraphQL。簡言之，GraphQL 賦予前端類似執行 SQL 查詢的能力，讓開發者能從後端獲取所需資料。透過這種方式，後端被抽象化為一個資料服務層，實現前後端開發的解耦。下文我將採用 Pattern 的結構，對此解決方案進行分析。

## Pattern 分析

### Context

開發團隊正在建構一個具備複雜領域模型的軟體系統，實體間存在多層次的關聯（如：使用者、訂單、產品、類別彼此交互參照）。後端關注業務流程與資料一致性，前端關注介面互動與使用者體驗，雙方的關注點不同。系統同時服務多樣化的消費端——Web 應用、行動裝置 App、甚至第三方合作夥伴——各消費端對同一份底層資料的需求，在結構與粒度上存在差異。目前系統透過以端點為單位的 API 架構（如 RESTful API）對外提供資料。

### Problem

如何讓多樣化的消費端，能夠從複雜的關聯式資料模型中，各自高效地取得所需的資料視圖？

### Forces

#### Force 1：資料的儲存結構與取用需求的錯配

後端依照領域邏輯與業務規則組織資料，按職責劃分邊界。資料源可能來自資料庫、外部服務或其他系統——其內部結構對消費端而言是透明的。然而，消費端需要的是依照各自情境組裝好的結構——一張卡片可能同時需要來自不同資料源的片段。後端的資料組織是為了維護一致性與業務正確性，消費端的資料需求是為了呈現與互動，兩者的組織邏輯存在結構性的錯配。當消費端的種類增加（Web、行動裝置、合作夥伴 API），每一種都需要不同的組裝結果，而為每種需求預先準備固定的組裝方式（如客製端點），其數量將隨消費端的多樣性持續增長。

#### Force 2：端點式架構的結構性限制

資料關聯在本質上是圖狀的（如：使用者 → 訂單 → 產品 → 類別），但以端點為單位的 API 架構將圖狀關聯切割為離散的存取點，每個端點僅回傳固定結構的樹狀資料。這導致兩個方向相反的問題同時存在：端點包含的欄位多於消費端所需，造成 Over-fetching；包含的欄位少於所需，消費端必須發出額外請求再自行組裝，造成 Under-fetching。端點的結構是在設計時決定的，但消費端的需求是在使用時才確定的，兩者之間的落差在固定結構下難以彌合。

#### Force 3：取用自由度與資料治理

若賦予消費端完全的資料存取自由（如直接查詢資料庫），後端將失去對資料一致性、安全性與領域邊界的治理能力。但若由後端全權決定資料的提供方式，消費端的每次需求變更都需等待後端調整，形成瓶頸。消費端的自由度與後端的治理能力是一組彼此拉扯的張力。

### Solution

引入一個**自助式資料存取層（Self-Service Data Access Layer）**，具體做法：

1. **由後端定義 Schema（型別系統）**：使用強型別描述所有資料類型與關聯關係。這份 Schema 作為雙方共享的契約，不僅反映了後端的領域模型，也將「定義資料邊界」的權利保留在後端。

2. **由前端撰寫 Query（查詢）**：前端透過宣告式語言定義所需的資料結構，發送至單一端點後，系統將自動組裝並回傳符合查詢定義的結果。「要什麼資料」的決定權交給前端。

3. **組裝邏輯自動化**：Resolver 層自動處理參照解析、關聯走訪、欄位篩選，無須再由人工為每種需求撰寫專屬的 endpoint。

**這個解決方案的本質是：在複雜的資料圖譜上建立一層型別化的自助存取協定。** Schema 將圖狀的資料關聯顯式化為型別系統，讓消費端能「看見」資料拓撲；Query 讓消費端以宣告式語法從這個拓撲中裁切出所需的視圖，精準到欄位層級；Resolver 將宣告式需求轉化為實際的資料存取操作，自動完成關聯走訪與組裝。

從組織層面來看，這相當於用一個技術契約（Schema）取代了反覆的人際溝通。原本每次消費端需要新的資料組合都需要跨團隊協調，現在轉化為消費端在契約範圍內的自助操作。團隊間的協調焦點從「每個畫面需要什麼資料」收斂為「Schema 如何演進」，溝通頻率與範圍隨之縮減。

### Resulting Context

#### 技術面效果

- **資料傳輸效率提升。** 單一請求即可取得所需欄位，緩解了 Over-fetching 與 Under-fetching 的問題。
- **多樣化的消費端不再需要客製端點。** 每個消費端透過 Query 自行定義所需的資料視圖，後端無須為不同平台維護不同的端點。
- **Schema 成為活的文件。** 所有消費端對「系統有哪些資料、如何關聯」有了結構化且可查閱的共同理解，減少了認知差異。

#### 組織面效果

- **前後端解耦從「流程層面」提升至「架構層面」。** 只要 Schema 涵蓋所需，消費端即可自行調整 Query，無須觸發後端的變更流程。
- **跨團隊溝通集中在 Schema 的演進上**——什麼時候新增類型、什麼時候廢棄欄位。這類變更的頻率遠低於個別畫面的需求變更，溝通頻率隨之降低。

#### 需要進一步處理的 Forces（可再套用其他 Pattern）

- **後端複雜度增加。** 前端的自由度意味著後端要能處理任意結構的查詢。
- **安全性與效能控制。** 查詢的決定權在前端，後端需要防範惡意或過度複雜的查詢。
- **快取策略需重新設計。** REST 的 URL-based 快取不再適用。
- **Schema 演進需要治理。** Schema 成了全系統的共享契約，它的變更影響所有消費者。需要 **Schema Governance** 的流程——版本管理、廢棄策略、向後相容性規範。這是一個新的組織問題，但其規模和頻率相對較低。
- **學習成本。** 團隊需要理解新的概念、工具鏈、除錯方式。短期內生產力會下降。

## 個人的見解

我認為 GraphQL 的核心價值，在於提供一個型別化的自助存取協定，讓多樣的消費端能從複雜的資料模型中各自高效地取得所需資料，同時保留後端的治理能力。

然而，對於前端 UI 變動頻率不高的專案，或是以 Feature Team 為核心組織架構的團隊，導入 GraphQL 的邊際效益未必顯著。
以我個人日常開發的製造業內部系統為例，多數專案由我一人同時負責網頁前端頁面與後端 Web API，系統的核心在於穩定地執行生產流程，並確保嚴謹的資料一致性。這類系統的 UI 呈現通常高度遵循標準作業流程 (SOP)，需求相對穩定且明確。

因此，在領域模型如此清晰的情境下，我傾向維持 RESTful API 架構，並採用 Contract-First（契約優先）的開發模式：初步定義 OpenAPI Specification 與資料庫 Schema 後，直接從規格生成帶有強型別的 TypeScript 介面 / JSDoc 與 Mock Service，供前端先行建構 UI。若開發過程中發現資料缺漏，我也能即時回頭修正 API 規格或調整資料表結構，迭代過程更為高效。

當專案規模擴大至多人協作時，若專案性質屬於 UI 需求相對固定、變動性低的類型，API 規格的異動頻率自然不高。在溝通成本尚能受控的前提下，維持現有的 RESTful 架構會更為務實，未必需要引入 GraphQL 來增加技術架構的複雜度。
