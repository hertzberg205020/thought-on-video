# 影片心得 (統一介面架構: 一個 API 統一前後端開發)

影片中雖未明言，但根據其描述的「統一介面架構」特性推測，其底層實作應是基於 GraphQL。簡言之，GraphQL 賦予前端類似執行 SQL 查詢的能力，讓開發者能從後端獲取所需資料。透過這種方式，後端被抽象化為一個資料服務層，實現前後端開發的解耦。下文我將採用 Pattern 的結構，對此解決方案進行分析。

## Pattern 分析

### Context

開發團隊正在建構一套基於多實體關聯領域模型的軟體系統，內部實體（如使用者、訂單、產品與類別）存在多層次的交互參照。由於前後端關注點不同——後端著重於業務邏輯與資料一致性，前端則專注於介面呈現與互動體驗——加上系統需同時服務 Web 應用程式、行動裝置 App 及第三方 API 等多種客戶端，導致各端對同一底層資料的結構與粒度需求皆不盡相同。目前，系統主要仍透過基於端點（Endpoint）的架構（如 RESTful API）來提供資料服務。

### Problem

如何讓不同的客戶端，從具備多實體關聯的領域資料模型中，有效率地取得各自所需的資料結構？

### Forces

#### Force 1：資料領域模型與視圖需求的結構性差異

後端依據領域邏輯與業務邊界組織資料，並隱藏底層資料來源（如資料庫或外部服務）的實作細節。相對地，客戶端基於使用者介面（UI）的呈現需求，經常需要在單一視圖中聚合多個資料源的特定欄位。由於後端資料結構以「維護業務規則與狀態一致性」為目的，而客戶端以「介面交互」為導向，兩者在資料組織原則上存在本質差異。此結構性差異導致資料的「聚合與重組」成為必然的處理環節，進而衍生出架構設計的決策節點：該組裝邏輯應配置於哪一系統層級執行，以及由哪一方定義資料聚合的規格。

#### Force 2：端點式 API 架構的限制

領域實體間的關聯本質呈現圖狀（Graph）結構，而基於端點（Endpoint）的 API 架構將資料存取點離散化，且各端點預設回傳固定結構的資料內容（Payload）。此架構特性在對應多變的客戶端需求時，會產生兩種現象：一為超取（Over-fetching），即端點回傳超出客戶端所需的冗餘欄位，增加網路頻寬消耗；二為少取（Under-fetching），即單一端點無法提供完整資訊，導致客戶端必須發起多次網路請求（Network Round-trips）並自行執行資料關聯。若為規避上述現象而替特定 UI 需求開發專屬端點，隨著客戶端種類與視圖版本的增加，API 的總量、版控複雜度與後續維護成本將隨之提升。

#### Force 3：存取彈性與系統管控的權衡

若開放客戶端過大的資料存取權限（例如直接查詢資料庫），後端將難以管控資料一致性、安全性與領域邊界。反之，若由後端完全主導資料的提供格式，客戶端的任何需求變更皆需依賴後端修改 API，容易形成開發瓶頸。因此，如何在客戶端的查詢彈性與後端的系統管控之間取得平衡，是架構設計上的挑戰。

### Solution

引入一個**自助式資料存取層（Self-Service Data Access Layer）**，具體實作機制如下：

1. **後端定義 Schema（資料結構與型別系統）**：透過強型別宣告所有資料實體與關聯性。此 Schema 作為前後端共享的 API 契約，不僅反映領域模型，亦由後端保有資料邊界與存取權限的定義權。

2. **前端定義 Query（動態查詢）**：前端透過宣告式語言（Declarative Language）撰寫查詢，依據介面需求定義所需的資料結構。發送至單一端點後，系統將回傳符合該查詢定義的結果。資料存取的粒度與組合由前端決定。

3. **資料解析與組裝自動化**：透過解析器（Resolver）層自動處理參照解析、資料關聯查詢與欄位篩選，後端無須為不同的前端需求開發或維護專屬的 API 端點。

此架構模式的核心在於：於複雜的關聯式資料模型之上，建立一層基於型別系統的動態存取介面。Schema 將資料的關聯拓撲明確化；Query 讓客戶端以宣告式語法精準提取所需欄位；Resolver 則負責將動態查詢轉換為實際的資料存取與組裝操作。

在開發流程上，此 API 契約（Schema）降低了跨團隊的溝通成本。以往客戶端每次需要新的資料組合時，皆須與後端協調開發新端點；導入此模式後，轉變為客戶端在 Schema 規範範圍內的自助式查詢。團隊協調的重點從「特定畫面所需的資料欄位」轉移至「Schema 的設計與演進」。

### Resulting Context

#### 技術面效果

- **資料傳輸效率提升：** 單一請求即可取得精確的所需欄位，有效解決前端 Over-fetching（超取）與 Under-fetching（少取）的問題。
- **提升客戶端開發彈性：** 各客戶端（如 Web、iOS、Android 及第三方夥伴）可透過 Query 自行定義所需的資料結構，後端無須為不同平台維護專屬的端點。
- **Schema 具備即時文件（Living Documentation）功能：** 客戶端皆可透過 Schema 查閱系統支援的資料型別與關聯，降低前後端對資料定義的認知落差。

#### 組織面效果

- **提升前後端架構解耦程度：** 只要前端的需求在既有 Schema 涵蓋範圍內，即可自行修改 Query 並部署，無須依賴後端調整 API 或重啟發布流程。
- **溝通成本收斂：** 跨團隊協作聚焦於 Schema 的演進（如新增型別或廢棄欄位）。這類核心領域模型的變動頻率，通常遠低於前端介面頻繁的迭代需求。

#### 需要進一步處理的 Forces（可再套用其他 Pattern）

- **後端系統複雜度增加：** 前端查詢的高度彈性，意味著後端必須具備處理動態且未知查詢結構的能力。
- **安全性與系統資源防護：** 由於查詢控制權轉移至前端，後端需實作查詢深度限制（Query Depth Limiting）或複雜度分析（Query Complexity Analysis），以防止惡意或過度消耗資源的查詢操作。
- **快取（Cache）機制重構：** 傳統基於單一 URL 的 HTTP 快取策略不再適用，系統需導入欄位層級（Field-level）或物件層級的快取機制。
- **Schema 治理（Governance）挑戰：** Schema 作為全局 API 契約，其變動將影響所有客戶端。團隊需要建立版本控制、欄位廢棄（Deprecation）策略與向後相容性規範。
- **技術導入與學習成本：** 團隊需熟悉新的技術堆疊、宣告式查詢概念與除錯工具，短期內可能面臨學習曲線與開發節奏的波動。

## 個人的見解

我認為 GraphQL 的核心價值，在於提供一個型別化的自助存取協定，讓不同的客戶端能從複雜的資料模型中各自高效地取得所需資料，同時保留後端的治理能力。

然而，對於前端 UI 變動頻率不高的專案，或是以 Feature Team 為核心組織架構的團隊，導入 GraphQL 的邊際效益未必顯著。
以我個人日常開發的製造業內部系統為例，多數專案由我一人同時負責網頁前端頁面與後端 Web API，系統的核心在於穩定地執行生產流程，並確保嚴謹的資料一致性。這類系統的 UI 呈現通常高度遵循標準作業流程 (SOP)，需求相對穩定且明確。

因此，在領域模型如此清晰的情境下，我傾向維持 RESTful API 架構，並採用 Contract-First（契約優先）的開發模式：初步定義 OpenAPI Specification 與資料庫 Schema 後，直接從規格生成帶有強型別的 TypeScript 介面 / JSDoc 與 Mock Service，供前端先行建構 UI。若開發過程中發現資料缺漏，我也能即時回頭修正 API 規格或調整資料表結構，迭代過程更為高效。

當專案規模擴大至多人協作時，若專案性質屬於 UI 需求相對固定、變動性低的類型，API 規格的異動頻率自然不高。在溝通成本尚能受控的前提下，維持現有的 RESTful 架構會更為務實，未必需要引入 GraphQL 來增加技術架構的複雜度。
